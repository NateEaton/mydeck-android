## Grouping rationale

Six slices, grouped by **kind of change** rather than by punchlist tier. This minimizes context-switching within each slice and keeps each agent in one mental model at a time. Slices are ordered so the highest-risk correctness fixes (containing the sole remaining blocker B2) land first; pure cleanup is last.

Items deferred (none in this plan): no item is high-complexity AND low-value. Everything is in scope.

1. **Slice 1** (sync resilience) — clears B2, the only remaining blocker.
2. **Slice 2** (Highlights repo) — tightens correctness before UX layer touches the same files.
3. **Slice 4** (WebView) — independent surface; could run parallel with 2 if you want.

## Execution mode

**Recommendation: separate threads per slice, not dispatched subagents.**

Rationale:
- Each slice requires `./gradlew :app:assembleDebugAll` + device install + manual UI testing that can't be automated from this thread.
- The CLAUDE.md anti-bloat protocol mandates sequential, scout-before-read, ≤3 files per research turn. Subagents don't honor that as cleanly as a fresh thread will.
- You want to validate between slices anyway. Threading them keeps your validation cadence in your hands.
- Each thread starts fresh with a tight prompt — no context drift from earlier slices.
