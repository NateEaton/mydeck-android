# Spec: Stable Debug Verification Tasks

## Summary

The repository instructions currently reference generic debug tasks like `:app:assembleDebug`,
`:app:testDebugUnitTest`, and `:app:lintDebug`. Those names are now ambiguous because the app has
multiple debug flavors (`githubReleaseDebug` and `githubSnapshotDebug`).

This spec proposes adding stable, human-friendly verification task names and updating agent/project
docs to reference them.

## Problem

- `:app:testDebugUnitTest` no longer resolves cleanly because Gradle now sees multiple matching
  flavor-specific tasks.
- Repository guidance in `AGENTS.md` and `CLAUDE.md` is therefore misleading.
- Humans and coding agents have to remember flavor-specific task names or discover them ad hoc.

## Goals

- Provide unambiguous debug verification commands.
- Keep the default verification workflow short and easy to remember.
- Preserve coverage across both debug flavors when desired.

## Non-Goals

- Changing product flavors or build types.
- Removing flavor-specific tasks generated by Android Gradle Plugin.
- Altering CI behavior unless explicitly opted into later.

## Proposed Approach

Add explicit aggregate Gradle tasks in `app/build.gradle.kts`:

- `assembleDebugAll`
- `testDebugUnitTestAll`
- `lintDebugAll`

Task behavior:

- `assembleDebugAll` depends on:
  - `:app:assembleGithubReleaseDebug`
  - `:app:assembleGithubSnapshotDebug`
- `testDebugUnitTestAll` depends on:
  - `:app:testGithubReleaseDebugUnitTest`
  - `:app:testGithubSnapshotDebugUnitTest`
- `lintDebugAll` depends on:
  - `:app:lintGithubReleaseDebug`
  - `:app:lintGithubSnapshotDebug`

## Documentation Changes

Update:

- `AGENTS.md`
- `CLAUDE.md`

New default verification text:

- `./gradlew :app:assembleDebugAll`
- `./gradlew :app:testDebugUnitTestAll`

If UI/resources/build config/dependencies changed:

- `./gradlew :app:lintDebugAll`

## Why This Approach

- Keeps docs stable even if flavor names change later.
- Avoids teaching every contributor the full flavor matrix for common local verification.
- Makes agent automation simpler and less error-prone.

## Alternative Considered

Use one concrete flavor in docs only:

- `:app:assembleGithubReleaseDebug`
- `:app:testGithubReleaseDebugUnitTest`
- `:app:lintGithubReleaseDebug`

Pros:

- Minimal implementation work.

Cons:

- Leaves docs coupled to a specific flavor.
- Makes it easier to miss a flavor-specific regression.

## Acceptance Criteria

- Running `./gradlew :app:assembleDebugAll` builds both debug flavors successfully.
- Running `./gradlew :app:testDebugUnitTestAll` runs both debug unit test suites successfully.
- Running `./gradlew :app:lintDebugAll` runs lint for both debug flavors successfully.
- `AGENTS.md` and `CLAUDE.md` reference only the new aggregate task names for default verification.

## Open Question

If the team wants a faster default loop, we can keep the aggregate tasks for CI/agents and also
document a “fast path” using only `githubReleaseDebug` for local iteration.
