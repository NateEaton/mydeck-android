### B2. `CancellationException` swallowed across multiple sync sites **[V]** — reproduced in logs 2026-05-13-210813
Original [V] finding pointed at `FullSyncWorker.kt:83-85`. Log review widened the scope: the same anti-pattern lives in `BookmarkRepositoryImpl.performFullSync()` and (probably) elsewhere on the sync path. Two `LoadBookmarksWorker` full syncs were cancelled mid-flight and both produced:
- `E/BookmarkRepositoryImpl$performFullSync: Full sync failed` + `JobCancellationException` (cancellation logged as error and converted to `SyncResult.NetworkError`)
- `W/...Failed to clear remote bookmark IDs ... JobCancellationException` (the `finally` block ran a suspending DB call on the already-cancelled coroutine, throwing again — staging rows leak)

Fixes:
1. `FullSyncWorker.doWork()` (`app/src/main/java/com/mydeck/app/worker/FullSyncWorker.kt:83-85`) — add `catch (e: CancellationException) { throw e }` before the broad catch.
2. `BookmarkRepositoryImpl.performFullSync()` (`app/src/main/java/com/mydeck/app/domain/BookmarkRepositoryImpl.kt:796`) — same `CancellationException` rethrow before the broad catch.
3. Same file, `finally` block at 799-804: wrap `clearRemoteBookmarkIds` in `withContext(NonCancellable) { … }` so staging-row cleanup actually runs on cancellation. This directly mitigates S14 (the log proves staging-row leakage is happening today, not just hypothetically on force-stop).
4. Audit `LoadBookmarksWorker.doWork()` and `BookmarkRepositoryImpl.performDeltaSync()` for the same pattern.

### ~~B3. `BookmarkScrollInterface.reportProgress()` crosses JS thread → Compose state without dispatch~~ — DOWNGRADED to Should-fix 2026-05-13
Stress-tested 2026-05-13 (open multiple long articles, scroll vigorously) — `mydeck-logs-2026-05-13-203702` shows no thread-violation exception. The callback writes to a `MutableState<Float>`; Compose's snapshot system is thread-safe for value writes, so the predicted `IllegalStateException` does not fire in practice. Still defensible defensive hygiene — see new S26.

## 🟡 Should-fix (fix before release; not necessarily before merge)

### Highlights core

**S1. `AnnotationSummaryDto.note: String` is non-nullable with no default** **[V]**
`AnnotationSummaryDto.kt:11`. Will throw `SerializationException` if the server ever omits the field. Change to `val note: String = ""` (or nullable).

**S2. ~~`remote_annotation_ids` lacks the `syncRunId` composite PK~~** — REPLACED 2026-05-13 with startup-cleanup approach
OpenAPI review (`docs/openapi-spec.json` lines 2938-3098): annotations have no delta endpoint — `GET /bookmarks/annotations` accepts only `limit`/`offset`, `GET /bookmarks/{id}/annotations` accepts nothing. Every reconciliation is a full scan, gated by process-global `refreshMutex.tryLock()`. Concurrent-run corruption can't happen in-process; the only remaining risk is force-stop/OOM staleness — same problem as S14 for bookmarks. Solve it the same way: add startup `DELETE FROM remote_annotation_ids` rather than a schema-16 composite-PK migration. Cheaper, no migration risk, symmetric with the S14 fix.

**S3. `force` ignored by per-bookmark backoff** **[R]**
`HighlightsRepository.kt:603-611`. Backoff check fires before the `if (force)` guard, so `USER_RETRY` routed through `requestBookmarkAnnotationChecks` doesn't bypass per-bookmark failure backoff. Spec requires manual retry to bypass throttles.

**S4. Duplicate `SCREEN_OPEN` refresh trigger** **[R]**
`HighlightsViewModel.kt:92-94` + `HighlightsScreen.kt:68-78`. Both `init` and `DisposableEffect`'s `ON_RESUME` fire `SCREEN_OPEN` within milliseconds. Remove the `init` call; rely on `ON_RESUME`.

**S5. `reconcileAllAnnotations` writes `lastGlobalAttemptAt` before `tryLock`** **[R]**
`HighlightsRepository.kt:358`. If `tryLock` fails, the attempt timestamp is already persisted even though nothing ran. Move the write inside the lock.

**~~S6. `clearFilters` does not reset `sortOrder`~~** — RESOLVED 2026-05-13: intentional. Sort and filter are separate concerns.

**~~S7. Inverted group layout: bookmark title rendered below its highlights~~** — RESOLVED 2026-05-13: intentional, matches Readeck UI.

### Sync infrastructure

**S13. Trust-the-server count can wipe local library** **[V]**
`BookmarkRepositoryImpl.kt:773-790`. `expectedTotalCount ?: 0` combined with `removeDeletedBookmars(syncRunId)`: if a proxy/cache returns `Total-Count: 0` on a non-empty account, every local bookmark is deleted. Add a defense-in-depth guardrail — refuse to delete >N% of local rows in one pass, OR hard-fail when expected count is 0 but local has rows.

**S14. Stale `remote_bookmark_ids` rows accumulate after process kill** **[V]** — confirmed via logs 2026-05-13-210813
`BookmarkRepositoryImpl.kt`. `finally`-block cleanup doesn't run on force-stop/OOM **and also doesn't complete on coroutine cancellation** (log shows `clearRemoteBookmarkIds` throwing `JobCancellationException` because the parent scope is already cancelled). The B2 fix (wrapping the `finally` cleanup in `NonCancellable`) handles the cancellation case; still add a startup cleanup (`DELETE FROM remote_bookmark_ids` is safe — table is always transient) for the force-stop/OOM case.

**S17. `scheduleBookmarkOrphanRepairFullSync` uses `KEEP`** **[R]**

**S18. `SettingsDataStoreImpl.clearCredentials()` / `saveCredentials()` skip 4 orphan-diagnostic keys** **[R]**

### WebView (downgraded from blocker)

**S27. Global highlights reconciliation backoff has no user-bypass path** **[V]** — discovered 2026-05-13 via logs `2026-05-13/MyDeckAppLog.0 3`
Repro: DNS lookup failed at offset=600 during MANUAL_SYNC annotation reconciliation. A single failure triggered a 5-minute backoff (`backoffUntil=lastFailure + 5min`). Within that window, every subsequent `APP_OPEN` and `SCREEN_OPEN` refresh request was correctly skipped — but **so would any user-initiated retry**, because no path is wired to bypass `backoffUntil`. User-visible consequence: Highlights screen displays 600 of 1797 highlights, no indication of partial state, no way to force a retry for 5 minutes.
Fix: route Highlights-screen pull-to-refresh (and any future "Retry" affordance) as `USER_RETRY` and bypass `backoffUntil` in `globalRefreshDecision`. Companion: surface the partial-cache state via the existing orphan-diagnostic banner plumbing (see N4) so the user knows highlights are mid-sync. Related to S3, which has the same pattern for per-bookmark backoff.

**S26. `BookmarkScrollInterface.reportProgress()` JS-thread → Compose state without dispatch** **[V → downgraded]**
`WebViewScrollBridge.kt:12-15` + `BookmarkDetailWebViews.kt:496-502`. `@JavascriptInterface` runs on the WebView JS thread; callback writes Compose state. No crash observed under stress (logs 2026-05-13-203702), because `MutableState` writes are snapshot-safe. Still inconsistent with `WebViewActionsBridge`, which wraps both methods in `Handler(Looper.getMainLooper()).post { … }`. Apply the same pattern for consistency and to insulate against future callback bodies that touch Views directly.

## Positive notes (out of context, but worth recording)

A few things this branch got noticeably right:

- Composite-PK isolation for `remote_bookmark_ids` is implemented end-to-end (entity, DAO, repo staging, count-mismatch abort, migration 14→15).
- Deletion-detection count mismatch returns `NetworkError` (retryable) rather than terminal `Error` — correct call.
- `removeDeletedBookmars` guards `isLocalDeleted = 0`, protecting pending-local-delete bookmarks from the full-sync sweep.
- Delta path correctly restricts per-bookmark annotation checks to `updatedIds` only — does not enqueue checks for every bookmark on full sync.
- `cacheComplete = true` is set unconditionally on success even when orphans were skipped — orphan repair is scheduled out-of-band.
- `BookmarkMetadataSyncCoordinator` is `@Singleton` and reaches both workers via Hilt's application-scoped component — gate is genuinely process-global.
- `upsertBookmark` (IGNORE + UPDATE) replacing `insertBookmark` (REPLACE) correctly prevents the SQLite DELETE+INSERT cascade from dropping `cached_annotation` rows.
- WebView Phase 5d cleanup is complete: `pendingAnnotationScrollId`, `scrollToAnnotation()`, `onAnnotationScrollHandled()`, and `getAnnotationViewportInfo()` are absent. The Channel-based `AnnotationRefreshEvent` replacement is a cleaner design.
- `WebViewScrollBridge.injectScrollManager()` is high-fidelity to the spec — idempotency guard, `initialPositionApplied` flip-on-success, annotation-id-over-percentage precedence, `behavior: 'instant'` for initial position.
- Note-edit DOM update uses Base64 encoding to sidestep JS string-injection concerns.
- `ContentPackageManager.updateCachedReaderHtml()` covers both full-package and Room text-cache modes from a single call site.
- 26 of 27 new strings are present in all 9 locale files (only `sync_content_waiting_network` was missed).
